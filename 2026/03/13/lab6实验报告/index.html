dup 函数会复制文件描述符，使新文件描述符指向与原文件描述符相同的资源。

考虑情景：

  • 父进程在 dup 中增加 p[0] 的 ref_count。
  • 父进程映射 newfd 到 p[0] 的文件对象。
  • 如果在增加 ref_count 后发生中断，调度到子进程。
  • 子进程在 read(p[0]) 中检查 p[0] 的 ref_count，如果此时 ref_count 是 2，可能错误地认为管道已被关闭，因为父进程还没有完成 dup 的全部操作。

Thinking 6.3 阅读上述材料并思考：为什么系统调用一定是原子操作呢？如果你觉得不是 所有的系统调用都是原子操作，请给出反例。希望能结合相关代码进行分析说明。

在 syscall 对应的异常处理程序 handle_sys 中，在处理系统调用之前，首先关闭外部中断（通过修改状态寄存器 (SR) 来实现），防止在系统调用执行期间发生中断，确保系统调用的原子性。在中断被禁用的情况下执行系统调用的内核代码。这保证了在系统调用执行过程中，不会有其他中断或异常打断当前的系统调用。

Thinking 6.4 仔细阅读上面这段话，并思考下列问题：

  • 按照上述说法控制 pipe_close 中 fd 和 pipe unmap 的顺序，是否可以解决上述场景的进程竞争问题？给出你的分析过程。
  • 我们只分析了 close 时的情形，在 fd.c 中有一个 dup 函数，用于复制文件描述符。 试想，如果要复制的文件描述符指向一个管道，那么是否会出现与 close 类似的问 题？请模仿上述材料写写你的理解。

可以解决。如果调换顺序，先解除 fd 的映射，再解除 pipe 的映射，那么 fd 的引用计数会先减少。即使在两个 unmap 操作之间发生中断，由于 pageref(pipe) > pageref(fd) 仍然成立，不会影响判断管道是否关闭的正确性。

在 dup 函数中也可能出现类似问题。如果 dup 的文件描述符指向管道，在增加引用计数和映射新文件描述符之间发生中断，可能导致pageref(pipe) == pageref(pipe[0])。

elf_load_seg 和 load_icode_mapper 函数的实现确保了 bss 段被正确加载到内存中：

elf_load_seg 函数：

  • 读取文件中每个段的数据（忽略 bss 段，因为它不占文件空间）。
  • 将读取的数据拷贝到内存中相应的虚拟地址。
  • 对于 bss 段，不需要读取文件数据，而是直接在内存中分配相应大小的空间，并初始化为 0。

load_icode_mapper 回调函数：

  • 在内核态分配虚拟内存，并将物理页映射到虚拟地址。
  • 保证 bss 段也得到正确处理，内存中相应位置被初始化为 0。

难点分析

本次实验的主要目的是通过实践深入理解操作系统的管道（pipe）机制，并学会利用该机制实现一个简单的shell。通过亲手编写代码，我们旨在将课堂上学到的理论知识与实际操作相结合，增强对操作系统内部工作原理的认识。

Shell的实现

这里我们调用 fork 复制了一个 Shell 进程，执行 runcmd 函数来运行命令，而原本的 Shell 进程则等待该新复制的进程结束。

对于新复制的 Shell 进程，我们查看一下 runcmd 函数，这个函数以及 parsecmd 函数是 Shell 的核心。首先调用一次 gettoken，这将把 s 设定为要解析的字符串。gettoken 函数似乎较为复杂，涉及到字符串的解析，就不深入了。

接着我们需要调用 parsecmd 将完整的字符串解析。解析的参数返回到 argv，参数的数量返回为 argc。

另外 parsecmd 还会返回 rightpipe。对于该值的理解需要从头到尾梳理一下 Shell 解析并执行命令的流程。首先为了方便讲述，我要定义三个名词。第一个是主 Shell 进程，即执行了 main 部分代码的 Shell 进程；第二个是执行命令的 Shell 进程，即调用 spawn 创建执行命令的进程；第三个是命令进程，即被 spawn 创建的进程。
